## 评审 1: 核心技术路线 — 最大风险点

### 🔴 关键发现：雀魂通信协议已被逆向

研究发现，雀魂（Mahjong Soul）底层使用 WebSocket + Protobuf 通信，社区已有成熟的协议逆向工程项目：

- **[Majsoul/proto](https://github.com/Majsoul/proto)** — 雀魂 Protobuf 协议定义
- **[Majsoul/majsoul_calc](https://github.com/Majsoul/majsoul_calc)** — 雀魂计算器（协议解析）
- **[kiruya302/mahjong_soul_union](https://github.com/kiruya302/mahjong_soul_union)** — 雀魂数据解析

**这意味着存在一条完全不依赖 CV 的路线**：直接拦截并解析游戏网络通信，获取 100% 准确的结构化数据。

### 对比分析

| 维度 | CV 识别方案 | 协议解析方案 |
|------|-----------|-------------|
| 准确率 | 97-99%（受画面干扰） | **100%**（原始数据） |
| 实现复杂度 | 高（训练、调优、维护） | **中**（解析 protobuf） |
| 抗游戏更新 | 差（UI 改版需重训模型） | **中**（协议变更需更新 proto 定义） |
| 性能开销 | 中（CPU/GPU 计算） | **极低**（纯文本解析） |
| 跨平台 | 需适配截图 API | **天然跨平台** |
| 反检测风险 | 无 | 低（只读取本地通信） |
| 可获取的信息 | 屏幕可见信息 | **全部信息**（含对手手牌等隐藏信息） |
| 开发周期 | 2-4 周 | **1-2 周** |
| 技术维护 | 持续（模型迭代） | **低**（proto 更新时调整） |

### 建议

**强烈建议采用协议解析方案作为主要路线，CV 方案作为备选或补充。**

理由：
1. 用户明确要求"准确性很重要"——协议解析天然 100% 准确
2. 无需标注数据、训练模型、维护检测管线
3. 获取的信息远超视觉可见范围
4. 性能开销接近零
5. 开发周期更短

**如果坚持 CV 方案，下面的评审仍然适用。**

## 评审 2: 数据合成可行性

### 🔴 风险：合成-真实域差距（Synthetic-to-Real Gap）

**问题**：
合成数据是将牌面素材粘贴到背景上，与真实游戏截图存在本质差异：

| 差异点 | 合成数据 | 真实截图 |
|--------|---------|---------|
| 牌面光照 | 无光照模型 | 有光照/阴影/反射 |
| 牌面边缘 | 锐利/规整 | 有抗锯齿、阴影、模糊 |
| 牌面重叠 | 无重叠 | 手牌有轻微重叠 |
| 背景 | 简单拼接 | 有粒子效果、桌面纹理 |
| 整体风格 | 拼贴感 | 一体的渲染结果 |

**后果**：纯合成数据训练的模型，在真实截图上的精度可能比合成验证集低 10-20 个百分点。

**缓解措施**：
1. **Domain Randomization**：大幅增加合成数据的多样性（光照、颜色、纹理随机化），让模型不依赖特定外观
2. **风格迁移**：使用 CycleGAN 将合成图转为游戏风格
3. **真实数据微调**：必须收集 200+ 张真实标注数据做 fine-tune，这是不可省略的步骤
4. **渐进式训练**：先合成数据预训练 → 再真实数据微调

### ⚠️ 风险：素材采集难度

从游戏中精确截取单张牌面并去除背景并非易事：
- 牌面有阴影和光照效果，难以干净去背景
- 不同状态下牌面外观不同（发光、高亮）
- 需要覆盖赤宝牌（红色）和普通牌两种外观

### 🔴 风险：横置牌和暗杠牌的训练数据

副露中的横置牌是一个特殊情况：
- 牌面旋转 90 度，但内容不变
- 需要作为旋转的牌面来检测（还是作为独立类别？）

**两种策略的权衡**：

| 策略 | 优点 | 缺点 |
|------|------|------|
| 旋转数据增强 | 不增加类别数 | 可能与正常牌混淆 |
| 独立类别 | 检测更准确 | 增加类别数，需更多训练数据 |
| 后处理旋转框 | 利用 YOLO 的旋转框检测 | YOLOv8 标准版不支持旋转框 |

**建议**：使用旋转数据增强，在训练时随机旋转部分样本 90 度，让模型学习旋转不变性。

## 评审 3: 区域定位可行性

### ⚠️ 风险：比例坐标的脆弱性

文档中定义的比例坐标是基于 1920x1080 分辨率的估计值。存在以下问题：

1. **Steam 版窗口模式**下，实际客户区尺寸可能不是 16:9（标题栏、边框占用）
2. **雀魂 UI 并非严格等比缩放**——研究报告指出，作为 Unity+CEF 应用，某些元素在不同分辨率下表现不同
3. **游戏更新可能改变布局**

**缓解措施**：
- 锚点定位必须作为主要方案，比例坐标只作为 fallback
- 提供校准工具：让用户在首次使用时框选各区域
- 从真实截图中测量准确的坐标值

### ⚠️ 风险：3D 透视变形

雀魂牌桌有轻微 3D 透视效果：
- 对面玩家的牌有上下方向的压缩
- 左右玩家的牌有左右方向的压缩
- 牌河远端的小牌检测精度天然较低

**缓解措施**：
- 对各区域分别做透视校正后再检测
- 牌河训练数据必须包含透视变形样本

## 评审 4: YOLOv8 类别设计问题

### ⚠️ 问题：40 类是否必要？

当前设计了 40 个类别，包括 `back`（牌背）、`rotated`（横置标记）、`dora_frame`（宝牌指示框）。

**质疑**：
- `back`（牌背）：对手手牌不需要识别具体内容，牌背检测的目的是什么？如果只是检测数量，模板匹配更简单。
- `rotated`（横置标记）：作为独立类别不合理——它不是一个牌面，而是牌面的一种状态。应通过旋转框或旋转增强处理。
- `dora_frame`（宝牌指示框）：同样不是牌面，应通过模板匹配或区域定位处理。

**建议**：
- 核心类别只保留 37 种牌面 + 3 种赤宝牌 = 40 类
- `back`、`rotated`、`dora_frame` 不作为 YOLO 类别，改用模板匹配或区域逻辑处理
- 但如果牌河中的横置牌需要识别具体是什么牌（而不仅仅是"这里有张横着的牌"），则需要在训练数据中包含旋转样本

### ⚠️ 问题：相似牌面的区分

以下牌面在视觉上高度相似，容易混淆：

| 易混淆组 | 原因 |
|----------|------|
| 一万 vs 二万 vs 三万 | 小数字差异，远距离更难区分 |
| 一筒 vs 二筒 | 图案差异细微 |
| 二索 vs 三索 | 竹子数量接近 |
| 白 vs 背面 | 白板几乎无图案 |

**缓解措施**：
- 增加这些类别的训练样本权重
- 在训练数据中增加难例挖掘（hard negative mining）
- 后处理规则：利用上下文排除不可能的牌面

## 评审 5: PaddleOCR 可行性

### 🟢 低风险

PaddleOCR 识别纯数字和有限字符集（东南西北一二三四）是已解决问题，精度接近 100%。

**唯一风险**：雀魂分数显示带有装饰效果（阴影、描边），可能影响 OCR。但通过预处理（灰度化、二值化、去噪）可以解决。

### ⚠️ 注意：分数变化动画

分数变化时有 +/- 数字弹出动画，此时 OCR 结果不可靠。需依赖帧状态检测跳过动画帧。

## 评审 6: 模板匹配可行性

### 🟢 低风险

动作按钮是固定 UI 元素，模板匹配精度高。

**唯一风险**：
- 按钮发光/闪烁效果会改变像素值
- 使用归一化相关系数法可缓解
- 连续 2 帧确认策略进一步降低误检率

## 评审 7: 性能预算

### 🟢 低风险

130ms 总预算 vs 200ms 目标，留有 70ms 余量。即使在性能较差的机器上，也不会严重超标。

**注意**：
- ONNX Runtime 首次推理有 warm-up 延迟（约 200-500ms），需要在启动时预热
- 多区域 YOLO 推理（手牌+4个牌河+宝牌+副露 = 7+次推理）可能超出 50ms 预算
  - 缓解：将多个区域拼接到一张大图上，只做一次推理

## 评审 8: 跨平台截图

### 🟢 低风险

`mss` 库跨平台截图成熟可靠，macOS 和 Windows 均有成熟的窗口管理 API。

**注意**：
- macOS 需要授予"屏幕录制"权限，需在文档中提示用户
- Windows 上 BitBlt 对硬件加速窗口可能截取到黑屏，需改用 Windows Graphics Capture API
- Steam 版雀魂基于 Unity + CEF，可能使用硬件加速渲染

## 评审 9: 遗漏的重要细节

### ⚠️ 遗漏 1：三人麻将对局

文档假设为四人麻将对局，但雀魂支持三人麻将（三人麻雀）。三人麻将的布局和规则不同：
- 无北风、无2-8万
- 布局只有三个方向（无对面）
- 需要区分四人和三人模式

### ⚠️ 遗漏 2：语言设置

雀魂 Steam 版支持多语言（英文、繁体中文、韩文等），UI 文字（如"吃""碰""杠"）的显示语言取决于游戏设置：
- 动作按钮文字可能是 "Chi" "Pon" "Kan" 而非中文
- 局次信息可能是 "East 1" 而非 "东一局"
- 需要支持多语言或限定中文

### ⚠️ 遗漏 3：游戏内 UI 元素遮挡

- 角色立绘可能遮挡牌河和手牌区域
- 对话气泡可能遮挡部分画面
- 需要建议用户在设置中关闭这些元素

### ⚠️ 遗漏 4：模型更新策略

雀魂不定期更新（新皮肤、新主题、UI 调整），每次更新可能影响：
- 牌面外观变化（新皮肤）
- 桌面背景变化（新主题）
- UI 布局微调
需要设计模型更新和反馈机制。

## 最终评审结论

### 核心问题：技术路线选择

**评审认为，CV 识别方案技术上可行，但不是最优选择。**

协议解析方案在所有关键指标上（准确率、性能、开发成本、维护成本）都优于 CV 方案。

### 如果选择 CV 方案的必要条件

1. **必须收集真实标注数据**（200+ 张），纯合成数据无法达到 97%+ 准确率
2. **必须处理合成-真实域差距**，domain randomization + 真实数据微调
3. **锚点定位优先于比例坐标**，比例坐标仅作 fallback
4. **首次使用需校准**，提供用户交互式的区域校准工具
5. **处理三人麻将和四人麻将两种布局**
6. **限定或支持多语言 UI**
