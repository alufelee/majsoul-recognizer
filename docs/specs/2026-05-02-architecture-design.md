# 雀魂麻将识别助手 — 架构设计文档

> 版本: 1.0 | 日期: 2026-05-02

## 1. 系统架构

### 1.1 整体数据流

```
┌──────────┐    ┌───────────┐    ┌──────────────────────────────────┐    ┌────────┐
│          │    │           │    │         识别引擎                  │    │        │
│  屏幕截图 │───►│  区域定位  │───►│  ┌─────────┐  ┌─────┐  ┌─────┐ │───►│ 结果输出│
│  Capture │    │  ZoneSplit│    │  │ YOLOv8   │  │ OCR │  │模板  │ │    │ Output │
│          │    │           │    │  │ 牌面检测  │  │文字  │  │匹配  │ │    │        │
└──────────┘    └───────────┘    │  └─────────┘  └─────┘  └─────┘ │    └────────┘
                                │         │        │        │      │
                                │         └────────┼────────┘      │
                                │                  ▼               │
                                │          ┌──────────────┐        │
                                │          │  规则校验引擎  │        │
                                │          │  Validator   │        │
                                │          └──────────────┘        │
                                └──────────────────────────────────┘
```

### 1.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
|------|------|------|------|
| Capture | 截取雀魂窗口画面 | 窗口句柄/标题 | BGR 图像 |
| ZoneSplit | 定位并裁剪各功能区域 | 完整截图 | 各区域图片切片 |
| TileDetector | YOLOv8 牌面检测 | 区域图片 | 牌面位置+类别+置信度 |
| TextRecognizer | PaddleOCR 文字识别 | 区域图片 | 文字内容+位置 |
| PatternMatcher | 模板匹配动作按钮 | 区域图片 | 匹配结果+位置 |
| Validator | 规则校验与时序融合 | 各模块原始结果 | 校验后的结构化数据 |
| GameState | 维护全局游戏状态 | 校验后数据 | 当前完整对局状态 |

## 2. Capture 模块

### 2.1 窗口发现

```
输入: 无
输出: 窗口句柄 + 窗口位置 + 窗口尺寸

流程:
1. 枚举系统所有窗口
2. 按标题匹配 "Mahjong Soul" 或 "雀魂"
3. 获取窗口位置和客户区尺寸
4. 返回窗口信息
```

**跨平台实现**：

| 平台 | 窗口枚举 | 截图方式 |
|------|---------|---------|
| Windows | `win32gui.EnumWindows` + `win32gui.GetWindowText` | `win32ui` BitBlt |
| macOS | `AppKit.NSWorkspace` + `NSRunningApplication` | `Quartz.CGWindowListCreateImage` |

**异常处理**：
- 窗口未找到 → 报错并等待重试
- 窗口被最小化 → 跳过截图，输出上一帧状态
- 窗口尺寸变化 → 触发 ZoneSplit 重新定位

### 2.2 截图策略

- 截图频率：默认 5 FPS，可通过配置调整（3-10 FPS）
- 只截取雀魂窗口客户区，不截全屏
- 截图后统一缩放到基准分辨率（1920x1080），消除分辨率差异
- 使用 `mss` 库作为跨平台截图的备选方案

### 2.3 帧状态检测

为避免在动画帧上做误检，增加帧状态判断：

```
静止帧判断逻辑:
1. 与上一帧做差分（SSIM 或像素差异率）
2. 差异率 < 阈值（如 1%）→ 判定为静止帧，可进行识别
3. 差异率 ≥ 阈值 → 判定为动画帧，跳过识别，沿用上一帧结果
```

**需要跳过动画的场景**：
- 摸牌滑入动画
- 出牌飞向牌河动画
- 分数变化弹出动画
- 和牌庆祝特效

## 3. ZoneSplit 模块

### 3.1 区域定义

以 1920x1080 基准分辨率定义各区域（比例坐标，实际使用时按窗口缩放）：

```
+--[Dora]---[Round Info]---[Remaining]--+
|                                        |
|  Left     Opponent       Right         |
|  Player   Discards      Player        |
|  Discards ┌──────────┐  Discards      |
|  + Calls  │  Central  │  + Calls      |
|           │  Table    │               |
|           └──────────┘               |
|                                        |
|        Self Discards                   |
|  [Calls] [==== Hand ====] [Actions]    |
|              [Timer]                   |
+----------------------------------------+
```

| 区域 | 比例坐标（x, y, w, h）| 说明 |
|------|----------------------|------|
| hand | (0.25, 0.82, 0.50, 0.12) | 手牌区域 |
| dora | (0.35, 0.02, 0.15, 0.06) | 宝牌指示器 |
| round_info | (0.42, 0.02, 0.16, 0.06) | 局次信息 |
| score_self | (0.35, 0.75, 0.12, 0.05) | 自家分数 |
| score_right | (0.85, 0.45, 0.10, 0.05) | 右家分数 |
| score_opposite | (0.42, 0.10, 0.12, 0.05) | 对家分数 |
| score_left | (0.05, 0.45, 0.10, 0.05) | 左家分数 |
| discards_self | (0.20, 0.68, 0.50, 0.10) | 自家牌河 |
| discards_right | (0.75, 0.25, 0.15, 0.35) | 右家牌河 |
| discards_opposite | (0.25, 0.18, 0.50, 0.10) | 对家牌河 |
| discards_left | (0.10, 0.25, 0.15, 0.35) | 左家牌河 |
| calls_self | (0.05, 0.82, 0.20, 0.10) | 自家副露 |
| actions | (0.70, 0.82, 0.25, 0.10) | 动作按钮 |
| timer | (0.45, 0.92, 0.10, 0.06) | 倒计时 |

> **注意**: 以上比例为初步估计值，需要从实际截图中校准。最终值应通过配置文件定义，方便调整。

### 3.2 锚点定位（备选/辅助方案）

比例坐标可能因 UI 版本更新而偏移，增加锚点定位作为辅助：

- 截取若干固定 UI 元素作为锚点模板（如宝牌区域的标签图标）
- 首次启动时用模板匹配找到锚点实际位置
- 根据锚点位置计算各区域坐标
- 缓存坐标直到窗口尺寸变化

### 3.3 分辨率适配

```
适配流程:
1. 截图后缩放到 1920x1080 基准分辨率
2. 在基准分辨率上进行区域分割
3. 所有后续识别统一在基准分辨率上操作
```

**为什么缩放而不是用比例坐标**：
- 统一分辨率后，模型只需训练一个尺度
- 模板匹配无需多尺度模板
- OCR 在固定分辨率下效果更稳定

## 4. TileDetector 模块（YOLOv8）

### 4.1 模型规格

| 参数 | 值 |
|------|-----|
| 模型版本 | YOLOv8-small（精度优先） |
| 输入尺寸 | 640x640 |
| 类别数 | 40 类（见下表） |
| 推理格式 | ONNX |
| 推理引擎 | ONNX Runtime（CPU） |

### 4.2 类别定义

| 编号 | 类别 | 编码 | 说明 |
|------|------|------|------|
| 0-8 | 万子 1-9 | 1m-9m | |
| 9-17 | 筒子 1-9 | 1p-9p | |
| 18-26 | 索子 1-9 | 1s-9s | |
| 27-30 | 东南西北 | 1z-4z | |
| 31-33 | 白发中 | 5z-7z | |
| 34-36 | 赤宝牌 | 5mr/5pr/5sr | 红色五万/筒/索 |
| 37 | 牌背 | back | 用于检测对手手牌和暗杠 |
| 38 | 横置牌标记 | rotated | 辅助检测横放的副露牌 |
| 39 | 宝牌指示框 | dora_frame | 辅助定位宝牌区域 |

### 4.3 检测策略

**手牌区域**：
- 放大到 1.5x 后检测，提高小牌精度
- 新摸的牌与手牌有间隔，检测后会自然分开
- 手牌按 x 坐标排序输出

**牌河区域**：
- 各家牌河分别裁剪后检测
- 按"先行后列、先左后右"的顺序排列输出
- 对面和侧面牌河有透视变形，训练数据需包含变形样本

**宝牌区域**：
- 单独检测，区域小精度高
- 数量动态变化（1-5 张），通过检测到的数量确定

**副露区域**：
- 需要同时检测牌面和横置状态
- 横置牌旋转 90 度，训练数据需包含旋转样本

### 4.4 置信度策略

| 置信度 | 处理方式 |
|--------|---------|
| ≥ 0.9 | 直接输出 |
| 0.7 - 0.9 | 输出但标记为 `low_confidence`，加入 warnings |
| < 0.7 | 不输出，帧间融合时沿用上一帧结果 |

## 5. TextRecognizer 模块（PaddleOCR）

### 5.1 识别对象

| 对象 | 内容 | 字符集约束 |
|------|------|-----------|
| 分数 | 纯数字 + 逗号 | `[0-9,]` |
| 局次-场风 | 单字 | `[东南]` |
| 局次-局数 | 中日文数字 | `[一二三四]` |
| 本场数 | 纯数字 | `[0-9]` |
| 供托棒 | 纯数字 | `[0-9]` |
| 倒计时 | 纯数字 | `[0-9]` |

### 5.2 优化策略

- **限定字符集**：每个识别区域配置独立的字符白名单，大幅减少误识别
- **预处理**：对分数区域做灰度化 + 二值化，去除装饰效果（阴影/描边）
- **后处理**：
  - 分数：移除逗号，转为整数
  - 局数：中日文数字映射为阿拉伯数字
  - 异常值过滤（分数不可能为负数，局数只能是 1-4）

### 5.3 OCR 区域定位

分数和局次信息位置相对固定，由 ZoneSplit 直接裁剪。但需注意：

- 分数变化动画期间 OCR 结果不稳定，依赖帧状态检测跳过动画帧
- 不同玩家的分数区域位置不同，需分别裁剪

## 6. PatternMatcher 模块

### 6.1 模板库

需要准备的模板：

| 模板 | 数量 | 说明 |
|------|------|------|
| 吃 | 1 | 按钮正常态 |
| 碰 | 1 | 按钮正常态 |
| 杠 | 1 | 按钮正常态 |
| 荣和 | 1 | 按钮正常态 |
| 自摸 | 1 | 按钮正常态 |
| 立直 | 1 | 按钮正常态 |
| 过 | 1 | 按钮正常态 |

### 6.2 匹配策略

```
匹配流程:
1. 从 actions 区域裁剪图片
2. 对每个模板做 matchTemplate（归一化相关系数法）
3. 匹配值 > 阈值（如 0.85）→ 判定该按钮存在
4. 收集所有存在的按钮作为可用动作列表
```

### 6.3 干扰处理

- 按钮发光/闪烁效果会导致模板匹配值波动
- 使用归一化相关系数法（`TM_CCOEFF_NORMED`）对亮度变化有一定鲁棒性
- 阈值设为 0.85（低于一般值），为闪烁留出容错空间
- 连续 2 帧检测到同一按钮才确认，避免单帧误检

## 7. Validator 模块

### 7.1 规则校验

```python
# 伪代码
class GameValidator:
    def validate(self, state: GameState) -> GameState:
        # 规则 1: 手牌数量校验
        hand_count = len(state.hand) + len(state.drawn_tile or [])
        calls_count = sum(len(call.tiles) for call in state.calls)
        expected = 13 + (1 if state.drawn_tile else 0) - (calls_count - state.kan_count * 3)
        if hand_count != expected:
            state.warnings.append(f"hand_count_mismatch: got {hand_count}, expected {expected}")

        # 规则 2: 牌数上限（每种牌最多4张）
        tile_counter = count_all_tiles(state)
        for tile, count in tile_counter.items():
            if count > 4:
                state.warnings.append(f"tile_overflow: {tile} appears {count} times")

        # 规则 3: 牌河连续性
        # 新检测到的牌河应只比上一帧多 0 或 1 张牌
        if self.prev_state:
            for player in ['self', 'right', 'opposite', 'left']:
                diff = len(state.discards[player]) - len(self.prev_state.discards[player])
                if diff < 0 or diff > 1:
                    state.warnings.append(f"discard_jump: {player} diff={diff}")

        # 规则 4: 分数合理性
        for player, score in state.scores.items():
            if score < 0 or score > 200000:
                state.warnings.append(f"score_anomaly: {player}={score}")

        return state
```

### 7.2 帧间融合

```
融合策略（滑动窗口 = 3 帧）:
1. 维护最近 3 帧的检测结果
2. 对每个检测目标，取 3 帧中出现次数最多的类别（多数投票）
3. 置信度取 3 帧的平均值
4. 如果 3 帧结果不一致，标记为 unstable
```

### 7.3 全局状态维护

维护一个持续更新的 `GameState` 对象，跨帧持久化：

- 只在帧状态检测判定为"静止帧"时更新
- 变化的字段更新，未变化的字段沿用
- 低置信度结果不覆盖高置信度历史值

## 8. 异常场景处理

### 8.1 游戏状态切换

| 场景 | 检测方式 | 处理 |
|------|---------|------|
| 进入对局 | 检测到手牌区域 | 初始化 GameState |
| 对局结束 | 检测到和牌/流局动画 | 冻结 GameState，等待下一局 |
| 界面切换 | 手牌区域消失 | 停止识别，等待游戏画面恢复 |
| 加载画面 | 窗口内容大面积纯色 | 跳过识别 |

### 8.2 识别失败处理

| 场景 | 处理 |
|------|------|
| 区域定位失败 | 使用默认比例坐标，标记 warning |
| 牌面检测无结果 | 沿用上一帧，连续 5 帧无结果则报错 |
| OCR 结果非法 | 沿用上一帧分数，标记 warning |
| 动作按钮全部匹配失败 | 不输出 actions，标记 "no_action_detected" |

### 8.3 干扰因素

| 干扰 | 严重度 | 处理策略 |
|------|--------|---------|
| 角色立绘遮挡 | 高 | 建议用户在设置中关闭立绘 |
| 牌桌主题变更 | 中 | 训练数据覆盖多种主题背景 |
| 桌面装饰/粒子效果 | 高 | 训练数据包含装饰效果样本 |
| 牌面发光（可操作牌） | 高 | 训练数据包含发光/非发光两种状态 |
| 3D透视变形（对面/侧面牌） | 中 | 训练数据包含透视变换增强 |
| 不同分辨率 | 低 | 统一缩放到基准分辨率 |
| JPEG压缩伪影 | 低 | 训练数据添加压缩增强 |

## 9. 多区域推理优化

### 9.1 区域拼接策略

直接对各区域分别推理（7+ 次 YOLO 前向）会超出 50ms 预算。优化方案：

**将多个区域拼接到一张 640x640 画布上，只做一次推理**：

```
┌────────────────────────────────────────┐
│  手牌区 (640x200)                       │
├──────────┬──────────┬──────────────────┤
│ 牌河-自  │ 牌河-对  │  宝牌区          │
│ (200x200)│ (200x200)│  (200x200)       │
├──────────┼──────────┤                  │
│ 牌河-左  │ 牌河-右  │                  │
│ (200x200)│ (200x200)│                  │
└──────────┴──────────┴──────────────────┘
         640 x 640 (单次推理)
```

- 每个区域有固定位置，推理后按位置归属拆分结果
- 只需 1 次 YOLO 前向传播，耗时约 30-50ms
- 牌面之间保持足够间距，避免跨区域误检

### 9.2 ONNX Runtime 预热

首次推理有 200-500ms warm-up 延迟。启动时执行：

```
1. 加载 ONNX 模型
2. 用随机数据做 3 次推理预热
3. 预热完成后再开始实际识别循环
```

## 10. 模型更新与反馈机制

### 10.1 游戏更新应对

雀魂不定期更新可能影响识别效果：

| 更新类型 | 影响程度 | 应对措施 |
|----------|---------|---------|
| 新牌面皮肤 | 高 | 素材重新采集 + 模型重训 |
| 新桌面主题 | 中 | 背景素材补充 + 数据增强 |
| UI 布局微调 | 低 | 锚点重新定位 |
| 新装饰效果 | 中 | 训练数据补充 |

### 10.2 错误反馈机制

- 运行过程中自动保存低置信度检测的截图（置信度 < 0.8）
- 保存到 `data/feedback/` 目录
- 用户可手动标记错误结果，用于后续模型微调
- 每次标记记录：截图路径、检测结果、正确结果、时间戳

### 10.3 模型版本管理

models/

├── v1.1/
│   └── ...
